# Objective:
Conduct an anatomy study under the iio_simple_dummy driver in the IIO subsystem with the goal to get an overview of drivers related to sensors. There are no practical tasks in this tutorial (only reading).

# Bullet Point Commentary (as I read the text, so expect problems):
* Sidenote: For the sake of learning I will be doing this segment in Portuguese, to avoid just reverberating what's already in the tutorial.

Ordering by code snippet:
* 1: Canal é a representação abstrata de um canal de dados, isto é, por onde o módulo recebe os dados de algum lugar, por exemplo de algum dispositivo. Nesse caso, temos um canal que recebe dados de voltagem e é o channel 0 (não entendi direito o propósito do index).
* 2: .info_mask_separate define quais atributos são únicos para esse canal. Conversando com o Gemini, aparentemente acontece alguma mágica que transforma esses bits em arquivos. Cada arquivo representando um atributo (como por exemplo o offset) e cada arquivo único para esse canal (se eu criar um outro canal de tipo voltagem ele não irá compartilhar esses arquivos especificados).
* 3: .info_mask_shared_by_dir irá definir uma mask no qual todos os canais com a mesma direção irão compartilhar. Não entendi exatamente o que mask quer dizer nesse caso.
* 4: Caso o canal utilize um buffer, .scan_index é utilizado como um identificador único para organizar este canal dentro do buffer. Já .scan_type define como estão organizadas as informações dentro do buffer.
* 5: Para tutoriais futuros.
* 6: Agora estamos olhando um novo canal, porém dessa vez é um canal do tipo voltagem e que tem uma configuração denominada "differential". 
* 7: Aparentemente, os dados desse canal podem ser divididos em uma parte positiva (channel=1) e em uma parte negativa (channel=2). Isso seria um único canal, que recebe múltiplas informações (não sei exatamente o que isso significa).
* 8: Ainda estou em dúvida se um atributo ser compartilhado significa somente que os outros canais podem acessá-los ou se todos utilizam do mesmo arquivo (também não sei se realmente são criados arquivos).
* 9: Um outro canal de aceleração.
* 10: Aqui o tutorial fala sobre alguma especificidade que ocorre quando .modified=1. Não me parece relevante, acredito que seja apenas um exemplo das possibilidades na criação de um canal.
* 11: O tutorial pula ṕara o código 12.
* 12: A função iio_dummy_read_raw le dados do iio_dummy_state e salva nos parâmetros *val e *val2, essa leitura fica protegida pelo mutex_lock.
* 13: Apenas o código dessa leitura. A parte interessante é que esse código usa a mask única de cada canal. Aparentemente, apenas a mask final precisa ser única, podendo haver repetição entre alguns bits (o bit IIO_CHAN_INFO_RAW está ativo nas masks dos dois canais).
* 14: Apenas o código de escrita (nada muito diferente de antes).
* 15: O código se refere a assinatura do dispositivo (não sei exatamente o que isso é).
* 16: O código aloca espaço na memória para o dispositivo (primeira tarefa da função probe).
* 17: Inicialização das variáveis da instância do dispositivo.
* 18: Registro do dispositivo (ou da instância do dispositivo?).
